As part of the work for that we have to get Apple to notarize the ML C++ executables.  One requirement for this is to sign them.  (Another is the directory structure change from elastic/ml-cpp#593, but that's not the direct cause of this problem.)  The problem is that on installation Homebrew rewrites the rpaths from relative to absolute, invalidating the signatures.  When Homebrew messes with the binary contents it should also really remove signatures.\r\n\r\nI guess the workaround until then is to manually remove the signatures - see https://reverseengineering.stackexchange.com/questions/13622/remove-code-signature-from-a-mac-binary\r\n\r\nOne way we could fix this in the future would be to change where in the release process the Homebrew artifacts are created.  I believe at the moment it's:\r\n\r\n```\r\nbuild unsigned tar.gz artifact -> sign -> notarize -> upload to download site\r\n                                                   \\> extract the bits required by Homebrew\r\n```\r\n\r\nBut if it were:\r\n\r\n```\r\nbuild unsigned tar.gz artifact -> sign -> notarize -> upload to download site\r\n                               \\> extract the bits
